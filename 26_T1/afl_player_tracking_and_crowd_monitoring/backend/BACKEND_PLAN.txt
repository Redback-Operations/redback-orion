API ENDPOINTS (Gateway)
-----------------------
Auth (public):
  POST /auth/register     — create new user account
  POST /auth/login        — returns JWT token
  GET  /auth/me           — returns current logged-in user info (protected)

Video Processing (protected — require JWT):
  POST /upload                  — accepts video, fans out to player + crowd in parallel
                                  Returns: { job_id: "abc123" }  ← immediate, non-blocking

  GET  /status/{job_id}        — frontend polls for job progress
                                  Returns: { status: "processing" }
                                        or: { status: "done", results: { player: {...}, crowd: {...} } }
                                        or: { status: "partial", results: { player: {...}, crowd: null },
                                              errors: { crowd: "crowd service failed" } }
                                        or: { status: "failed", error: "..." }

  GET  /jobs?page=1&limit=10   — returns paginated past jobs for the logged-in user
                                  Returns: { total, page, limit, jobs: [ { job_id, status, created_at, results }, ... ] }

  GET  /jobs/{job_id}          — returns full details of a single past job
                                  Returns: { job_id, status, created_at, results: { player, crowd } }

  POST /jobs/{job_id}/retry    — re-runs only the failed service(s) for a partial job
                                  Does NOT re-run successful services
                                  Returns: { job_id, status: "processing" }

  DELETE /jobs/{job_id}        — delete a job and its results
                                  User: own jobs only | Admin: any job
                                  Returns: { message: "job deleted" }

System:
  GET  /health           — checks if player + crowd services are reachable
                           Returns: { gateway: "ok", player_service: "ok", crowd_service: "ok" }

STORAGE STRATEGY
----------------
Phase 1-3 (Local Development):
  - Uploaded video → saved to backend/uploads/ folder (deleted after processing)
  - Job status + results → stored in PostgreSQL
  - Player/crowd services read video via file path passed in the request

Phase 4 (Docker):
  - Uploaded video → saved to a shared Docker volume (/uploads/)
  - All services (gateway, player, crowd) mount the same volume
  - No need to send video over HTTP — just pass the file path
  - Job status + results → still in PostgreSQL (also runs in a container)


DATABASE SCHEMA (PostgreSQL)
-----------------------------
Table: users
  user_id      UUID        PRIMARY KEY
  email        VARCHAR     UNIQUE, NOT NULL
  username     VARCHAR     UNIQUE, NOT NULL
  password     VARCHAR     hashed password (bcrypt)
  role         VARCHAR     "admin" / "user"
  created_at   TIMESTAMP   when account was created

Table: jobs
  job_id        UUID        PRIMARY KEY
  user_id       UUID        FOREIGN KEY → users.user_id
  status        VARCHAR     "processing" / "done" / "failed"
  video_path    VARCHAR     path to uploaded video file
  player_result JSONB       results from player service
  crowd_result  JSONB       results from crowd service
  error         VARCHAR     error message if failed
  created_at    TIMESTAMP   when job was created
  updated_at    TIMESTAMP   last updated


AUTHENTICATION
--------------
- Users register and log in via the gateway
- Passwords are hashed with bcrypt (never stored in plain text)
- On login, gateway issues a JWT token
- All protected endpoints require the JWT token in the request header
- JWT token contains: user_id, email, role, expiry

Auth Endpoints:
  POST /auth/register  — create new user account
  POST /auth/login     — returns JWT token
  GET  /auth/me        — returns current logged-in user info

Protected Endpoints (require JWT):
  POST /upload                 — any logged-in user
  GET  /status/{job_id}       — own jobs (user) / all jobs (admin)
  GET  /jobs?page=1&limit=10  — own jobs (user) / all jobs (admin)
  GET  /jobs/{job_id}         — own jobs (user) / any job (admin)
  POST /jobs/{job_id}/retry   — own jobs (user) / any job (admin)
  DELETE /jobs/{job_id}       — own jobs (user) / any job (admin)
  GET  /health                — admin only

Tech used for Auth:
  - bcrypt (python-bcrypt)   — password hashing
  - JWT (python-jose)        — token generation and validation

  LoginRequest:      { email, password }

  UserResponse:      { user_id, email, username, role, created_at }

Upload:

  JobDetail:         { job_id, status, created_at, updated_at,
                       results: { player, crowd }, errors: { player, crowd } }
  JobListResponse:   { total, page, limit, jobs: [ JobSummary ] }

Health:
  HealthResponse:    { gateway, player_service, crowd_service }

----------------
We test the gateway WITHOUT needing the player or crowd services to be running.
We use mock responses to simulate what those services would return.

Tools:
  - pytest         — test runner
  - httpx          — async test client for FastAPI
  - pytest-asyncio — async test support

Test structure:
  backend/
  └── tests/
      ├── test_auth.py     — register, login, token validation
      ├── test_upload.py   — file upload, format validation, job creation
      ├── test_jobs.py     — job status, job list, pagination, retry
      └── test_health.py   — health check

What we mock:
  - Player service response → return fake player metrics JSON
  - Crowd service response  → return fake crowd metrics JSON
  - This lets us test the full gateway logic without running ML models

When to run tests:
  - Phase 1: test auth endpoints
  - Phase 2: test upload, jobs, retry with mocked services
  - Phase 3: remove mocks, test against real services

ROLE-BASED ACCESS CONTROL (RBAC)
---------------------------------
Two roles: "user" (default) and "admin"

Permissions:
  Endpoint                        | User              | Admin
  --------------------------------|-------------------|------------------
  POST /auth/register             | Yes               | Yes
  POST /auth/login                | Yes               | Yes
  GET  /auth/me                   | Yes (own)         | Yes
  POST /upload                    | Yes               | Yes
  GET  /status/{job_id}           | Own jobs only     | All jobs
  GET  /jobs                      | Own jobs only     | All users' jobs
  GET  /jobs/{job_id}             | Own jobs only     | Any job
  POST /jobs/{job_id}/retry       | Own jobs only     | Any job
  DELETE /jobs/{job_id}           | Own jobs only     | Any job
  GET  /health                    | No                | Yes

Implementation:
  - Role is stored in the users table (role: "user" / "admin")
  - Role is included in the JWT token payload
  - Two FastAPI dependencies:
      get_current_user   — any logged-in user
      require_admin      — admin only (returns 403 if not admin)
  - Default role on register: "user"
  - Admin accounts created manually in the database

New endpoint added:
  DELETE /jobs/{job_id}  — delete a job and its results
                           User can delete own jobs only
                           Admin can delete any job


BACKGROUND PROCESSING
---------------------
Video processing takes a long time (1-5 minutes). The gateway must NOT block
while waiting for player and crowd services to respond.

How it works:
  1. POST /upload is called
  2. Gateway saves video to uploads/ folder
  3. Gateway creates job in PostgreSQL (status: "processing")
  4. Gateway returns { job_id } IMMEDIATELY — does not wait
  5. Background task starts:
       → calls player + crowd services in parallel (asyncio.gather)
       → handles partial results if one service fails
       → updates job in PostgreSQL (status: "done" / "partial" / "failed")
  6. Frontend polls GET /status/{job_id} until status is no longer "processing"

Implementation:
  - Uses FastAPI built-in BackgroundTasks (no extra tools needed)
  - No message queue required for this project scope

  Example:
    @app.post("/upload")
    async def upload(file, background_tasks: BackgroundTasks):
        job_id = create_job_in_db()
        background_tasks.add_task(process_video, job_id, file_path)
        return { "job_id": job_id }  ← returns instantly

  process_video() runs in the background:
    → calls player + crowd in parallel
    → updates PostgreSQL when done


ERROR HANDLING STRATEGY
-----------------------
When one service fails and the other succeeds:
  - Job status is set to "partial" (not "failed")
  - Successful results are saved and returned to the frontend
  - Failed service is recorded in the errors field
  - Frontend shows available results + a Retry button

When user clicks Retry:
  - POST /jobs/{job_id}/retry is called
  - Gateway re-runs ONLY the failed service (not the successful one)
  - Video file path is retrieved from the job record in PostgreSQL
  - Job status updated to "processing" then "done" or "partial" again

Job status values:
  "processing" — job is currently running
  "done"       — both services completed successfully
  "partial"    — one service succeeded, one failed (retry available)
  "failed"     — both services failed

Video file size limit:
  - To be confirmed with player and crowd teams
  - Will be set as a config value in .env (MAX_UPLOAD_SIZE)

Video format validation:
  - Only accepted formats: .mp4, .avi, .mov
  - Gateway checks the file extension AND MIME type before processing
  - If invalid format → return 400 Bad Request immediately (do not call model services)
  - Accepted MIME types: video/mp4, video/x-msvideo, video/quicktime
